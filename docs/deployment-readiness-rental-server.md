# LoL診断サイト デプロイ判断とレンタルサーバー公開計画

最終更新: 2026-04-16

> 前提: 本ドキュメントは現時点のリポジトリ構成（Next.js App Router, `next build` / `next start` 前提）を基にした実務向けの判断メモです。

## 全体サマリー

- **結論**: 「今すぐ公開」は可能だが、**本番運用としては“条件付きGo”**。最低限の本番設定（本番URL固定、robots/sitemap、障害時運用、環境変数棚卸し）を整えてから公開すべき。
- **最も現実的な方式**: 初回は **Node.js実行対応レンタルサーバー（またはPaaS寄りマネージド）で `next build` + `next start`**。VPSは自由度が高いが、運用経験が浅い段階では事故率が上がる。
- **共有レンタルサーバーしか使えない場合**: `output: 'export'` で静的化すれば公開自体は可能。ただし、**結果URLごとの動的OGPは原則不可**。静的OGPや「結果ページnoindex」前提で割り切る。
- **公開前の最低ライン**: 本番ビルド成功、環境変数整理、OGPと共有URLの実地確認、404/500系導線、ドメイン・SSL、ログ確認手段、再デプロイ・ロールバック手順の明文化。

## Step 1. 今の状態でデプロイしてよいかの判断

| 項目 | 判定 | 理由 | 優先度 |
|---|---|---|---|
| 1. 現在の到達点の要約 | ほぼ到達 | MVP〜β準備の機能要件（トップ/診断/結果/共有/OGP/計測）が揃っている想定。 | 中 |
| 2. デプロイしてよい状態かどうか | 条件付きで可 | 技術的には公開可能。ただし本番運用に必要な設定・監視・運用手順が未確定だと障害時復旧が遅れる。 | 高 |
| 3. まだ不足しているもの | 要対応 | robots/sitemap、本番ドメイン確定、環境変数の本番値、障害時切り戻し手順、最低限の法務表記の確認が不足しやすい。 | 高 |
| 4. デプロイ前に最低限やるべきこと | 必須 | 本番ビルド検証、URL共有/OGP確認、Node実行可否確認、ログ確認導線、SSLとDNS設定を先に確定する必要がある。 | 高 |
| 5. 今デプロイする場合のリスク | 中〜高 | Node非対応サーバー選定ミス、環境変数漏れ、OGP崩れ、障害時ログ不足、ロールバック不能。公開後の信用毀損に直結。 | 高 |

### 不足の具体例（最低限）

- 本番URLを `NEXT_PUBLIC_SITE_URL` に固定し、全ページで同じ基準URLになること。
- `robots.txt` / `sitemap.xml` の最低限対応（特に結果ページnoindex方針の明文化）。
- 障害時に誰が何分以内に何を見るか（ログ、プロセス死活、直近デプロイ差分）。
- 利用規約・プライバシーポリシーの掲載要否判断（計測イベントがあるため実質必要）。

## Step 2. レンタルサーバーでのデプロイ方式比較

> 仮定: 日本国内の一般的なレンタルサーバー（共有/Node対応/VPS）を想定。

### 1) 共有レンタルサーバー（静的公開中心）

- 概要: FTP/SFTPでHTML/CSS/JSを配置する静的ホスティング中心。
- このプロジェクトとの相性: **中（静的化できる範囲のみ）**。
- メリット:
  - 安価、契約と運用が簡単。
  - セキュリティ運用をサーバー側に任せやすい。
- デメリット:
  - Node実行不可が多く、Next.js App Routerの動的機能と相性が悪い。
  - 動的OGP・SSR・API Routeがほぼ使えない。
- 向いているケース:
  - 静的LP/静的診断（クライアント完結）を最優先。
- 向いていないケース:
  - SSR必須、動的メタデータ、サーバー側計測、将来拡張。
- 運用難易度: 低。
- コスト感: 低。
- おすすめ度: ★★☆☆☆（Node不要に割り切るなら可）。

### 2) Node.js実行対応レンタルサーバー

- 概要: レンタルサーバー上でNodeアプリ常駐（またはプロセスマネージャ）運用。
- このプロジェクトとの相性: **高**。
- メリット:
  - Next.js標準運用（`next build` + `next start`）に近い。
  - App Router / 動的メタデータ / SSRを活かせる。
  - VPSより運用負荷が低い。
- デメリット:
  - プランによってNodeバージョン制限、プロセス制限、再起動制限がある。
  - ログや監視が弱い事業者もある。
- 向いているケース:
  - 初公開、少人数運用、将来SSR拡張の可能性あり。
- 向いていないケース:
  - 高トラフィック、大規模監視、細かいOS制御が必要。
- 運用難易度: 中。
- コスト感: 低〜中。
- おすすめ度: ★★★★☆（今回の第一候補）。

### 3) VPS

- 概要: 仮想サーバーを借りてOSレベルから自由に構築。
- このプロジェクトとの相性: **高（技術的には最良に近い）**。
- メリット:
  - Node/Nginx/PM2/監視などを自由設計できる。
  - 将来的な拡張（API・DB・バッチ）に強い。
- デメリット:
  - セキュリティ更新、ミドルウェア保守、障害対応が自己責任。
  - 本番運用経験が浅い場合はヒューマンエラーが増えやすい。
- 向いているケース:
  - インフラ管理ができる担当者がいる。
- 向いていないケース:
  - まず公開して運用を簡素化したい段階。
- 運用難易度: 高。
- コスト感: 中。
- おすすめ度: ★★★☆☆（将来の第二段階）。

### 4) コンテナ/Docker利用前提構成

- 概要: Dockerイメージ化して、対応環境（VPSやマネージド）に配置。
- このプロジェクトとの相性: **中〜高**。
- メリット:
  - 環境差異が減る、再現性が高い、移行しやすい。
- デメリット:
  - 共有レンタルサーバーではほぼ非対応。
  - Docker理解・CI整備が必要。
- 向いているケース:
  - チームで継続運用し、環境差異を最小化したい。
- 向いていないケース:
  - 最短公開が目的で、運用経験が浅い。
- 運用難易度: 中〜高。
- コスト感: 中。
- おすすめ度: ★★★☆☆（将来の運用基盤として有効）。

## Step 3. このプロジェクトに推奨するデプロイ方式

1. 推奨方式
   - **Node.js実行対応レンタルサーバーで `next build` + `next start` 運用**。

2. 推奨理由
   - 現在のNext.js構成を大きく変えず公開できる。
   - 共有サーバー静的化より機能制約が少ない。
   - VPSより運用リスクを抑えやすい（経験浅い前提に合う）。

3. 代替案
   - 代替A: 共有レンタルサーバー + 静的書き出し（機能妥協あり）。
   - 代替B: VPS + PM2 + Nginx（自由度優先）。

4. 採用判断の基準
   - Node 20+ が使えるか。
   - 常駐実行と再起動手段があるか。
   - 独自ドメイン + 無料SSL + ログ閲覧ができるか。
   - 月額と運用時間が許容内か。

5. この方式で注意すべきこと
   - Nodeバージョン固定（`.nvmrc` 推奨）。
   - プロセス死活監視（最低でも再起動手順明文化）。
   - `NEXT_PUBLIC_SITE_URL` など環境変数のデプロイごとの整合。

## Step 5. レンタルサーバー前提の実装・設定見直し

| 観点 | 現状想定 | 問題になりうる点 | 推奨修正方針 | 影響範囲 | 優先度 |
|---|---|---|---|---|---|
| Next.js の出力方式 | `next start` 前提 | Node非対応環境だと動かない | まずNode対応環境を前提化。代替として `output: 'export'` 分岐案を準備 | `next.config.ts`, ルーティング | 高 |
| 静的書き出しへの寄せ | 未実施 | 動的機能が残ると export 失敗 | 共有サーバーfallback用に静的化可否を棚卸し | 全体 | 中 |
| SSR必要性 | 結果ページは検索クエリで結果生成 | SSR不要でも実装は動くがOGP動的化要件で詰まる | 「動的OGP必要ならNode運用」「不要なら静的OGP」で方針固定 | `app/result/page.tsx` | 高 |
| OGP実装方式 | metadataあり、画像固定未明確 | SNSで画像なしだとCTR低下 | `public/ogp.png` を用意し `openGraph.images` を設定 | `app/layout.tsx`, `public/` | 中 |
| URL共有方式 | クエリ文字列共有 | URL長・改ざん・将来互換の課題 | 既存方式維持 + バージョン管理 + 不正時導線強化 | `lib/share.ts`, `app/result/page.tsx` | 中 |
| 環境変数 | `NEXT_PUBLIC_SITE_URL` 依存 | 未設定時localhost化する | 本番で必須化し、未設定時はビルド失敗させる選択を検討 | `app/layout.tsx`, `app/result/page.tsx` | 高 |
| 計測イベント送信 | `dataLayer/gtag` 最小実装 | 同意未取得時送信や重複送信 | 同意フラグ・重複抑止・イベント命名表を追加 | `lib/analytics.ts` | 中 |
| 画像/アセット配置 | 明示なし | OGP画像やfavicon不足 | `public/` に本番アセット一式を配置 | `public/` | 中 |
| Node不可時の代替 | 未整備 | 公開不能リスク | 静的専用ビルド手順（export）を別ドキュメント化 | `README` / docs | 高 |

## Step 6. 実際のデプロイ手順（推奨方式）

> 前提: Node.js対応レンタルサーバー、GitまたはSFTP配置可能、独自ドメインあり。

1. サーバー契約前に確認
   - Node.js 20+ 対応有無
   - 常駐プロセス運用可否（再起動方法含む）
   - SSL自動更新の有無
   - ログ閲覧方法

2. 契約後に確認
   - SSH接続可否
   - デプロイ先ディレクトリ
   - 公開ポート/リバースプロキシ仕様

3. Node.js / 実行環境確認
   ```bash
   node -v
   npm -v
   ```

4. ソースコード配置方法（例: git pull）
   ```bash
   cd /var/www/lol-type-choice
   git clone <REPO_URL> .
   ```

5. 依存パッケージインストール
   ```bash
   npm ci
   ```

6. build
   ```bash
   npm run build
   ```

7. 起動方法（暫定）
   ```bash
   PORT=3000 npm run start
   ```
   - 本運用はPM2等のプロセスマネージャ利用推奨。

8. ドメイン設定
   - DNSでA/CNAMEをサーバーへ向ける。
   - TTLは初期短め（例: 300秒）→安定後に延長。

9. SSL設定
   - サーバー管理画面またはACME連携で証明書発行。
   - HTTP→HTTPSリダイレクト有効化。

10. 公開確認
   - `/`, `/diagnosis`, `/result?r=...` をPC/スマホで確認。
   - 共有URLをSNSプレビューで確認。

11. ログ確認
   - アプリログで500系エラー確認。
   - アクセスログで404急増有無を確認。

12. 再デプロイ手順
   ```bash
   git fetch --all
   git checkout main
   git pull
   npm ci
   npm run build
   # プロセス再起動（例: pm2 restart app）
   ```

13. 障害時の切り戻し方針
   - 直前の安定コミットへ戻す。
   ```bash
   git log --oneline -n 5
   git checkout <stable_commit>
   npm ci
   npm run build
   # プロセス再起動
   ```
   - 重大障害時はメンテナンス文言をトップに表示。

## Step 7. 共有レンタルサーバーしか使えない場合の対応

1. 静的サイト化できる範囲
   - トップ、診断UI、結果表示（クライアント内計算）
   - 基本OGP（固定）

2. 静的化が難しい機能
   - リクエストごと動的OGP画像
   - サーバー側署名検証・サーバーログ連動機能

3. 代替実装案
   - `next export` 相当で配布。
   - 結果はクエリのみで再計算、OGPは固定1枚。
   - 計測はクライアント送信に限定。

4. どこまでなら妥協可能か
   - 「診断体験と共有URL再現性」を優先し、OGPの個別最適化は後回し。

5. 本番公開として成立するか
   - **成立はする**。ただし「高機能な動的演出」は制限される。

## Step 8. 運用の最低限設計

- デプロイ手順の定型化
  - `deploy.md` を作り、誰でも同手順で再現できるようにする。
- バックアップ
  - 少なくともソースはGitでタグ運用（`release-YYYYMMDD`）。
- ログ確認
  - 毎日1回は5xx/4xx急増を確認。
- 障害時対応
  - 「検知→切り戻し→告知→原因分析」の順序を固定。
- 環境変数管理
  - `.env.example` を最新化し、実値はサーバー側で管理。
- ドメイン更新
  - 年次更新期限をカレンダー登録。
- SSL更新
  - 自動更新失敗時の手動更新手順を一度検証。
- データ更新
  - チャンプ情報更新頻度（月1など）を決める。
- FAQ/文言更新
  - 非公式表記・免責・問い合わせ先の更新担当を決める。

## Step 9. このプロジェクト用の推奨公開手順まとめ

### まずやること
- ホスティング方式を「Node対応レンタルサーバー」に確定。
- 本番URL・環境変数一覧・最低運用ルールを確定。

### 次にやること
- ステージング相当で `npm ci && npm run build && npm run start` を通す。
- OGP・共有URL・エラーハンドリングを実機確認。

### 公開直前にやること
- DNS/SSLを有効化。
- robots/sitemap/metadata最終確認。
- 障害時の切り戻しコマンドを担当者間で共有。

### 公開後にやること
- 初週は毎日ログ確認。
- 404/500傾向と離脱イベントを見て優先改善。
- リリースタグ運用を開始。
